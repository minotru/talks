Operational Friction

Why naive systems fail

Checkpoint reloads are too slow

Huge model states turn disk I/O into the pacing item for the loop.

Training and inference are asynchronous

They run at different speeds, so restart-based coordination wastes GPUs.

Caches become stale after updates

KV state from old weights is a correctness bug, not just a performance detail.

Rollouts need structured metadata

Text alone is not enough when generation outputs become training artifacts.

1. Runtime Weight Updates

Avoid checkpoint reload
as the control plane

  • Checkpoint reloads are too slow for iterative RL.
  • We synchronize weights directly into the running inference cluster.
  • The important outcome is keeping GPUs generating.

2. KV Cache Correctness

Cache invalidation becomes
a correctness primitive

After a model update, prefix cache entries were created by a different policy. Keeping them is wrong, not merely suboptimal.

3. Lifecycle Coordination

Pause and resume
inference explicitly

  • We need consistent synchronization points for training stages.
  • Restarting servers to achieve that is too expensive.
  • A small lifecycle API changes the orchestration model completely.

Boundary Of Responsibility

What vLLM gave us
vs what we added

vLLM gave us

  • High-throughput generation
  • Continuous batching and scheduling
  • Paged attention and memory efficiency
  • A production-grade serving runtime
  • Useful generation outputs like logprobs

We added

  • Live weight synchronization into workers
  • Pause and resume lifecycle control
  • Prefix cache reset semantics
  • Rollout-oriented metadata plumbing
  • Integration with the RL control plane

Lessons Learned

RL stresses
infrastructure assumptions

  • Weight sync becomes part of the critical path.
  • Cache invalidation is a correctness problem.
  • Lifecycle APIs beat restart-based orchestration.
  • Training and inference infra are converging.
