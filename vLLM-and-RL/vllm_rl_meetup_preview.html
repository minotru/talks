Why Naive Breaks

Checkpoints are too large to route through disk every iteration.

Huge write + read path

Training writes the checkpoint, then inference reads it back.

Repeated startup cost

Inference reload becomes part of the training clock.

Fresh weights are required

Rollouts must reflect the current or near-current policy.

Concrete checkpoint size GPT-OSS-120B
120B params = ~240 GB weights in FP16 That full weight set has to move before inference can generate from the updated policy.
~240 GB checkpoint write
~240 GB checkpoint read

About ~480 GB of disk traffic per refresh, before reload/startup overhead.

Sync Loop: Keep vLLM Alive

Replace checkpoint reload with live weight transfer.

persistent Trainer

Updates policy weights after each batch.

InfiniBand + NCCL push updated weights
old path disk checkpoint reload is no longer in the hot loop
persistent vLLM

Serves rollouts from the updated model without restart.

trainer stays up vLLM stays up fresh rollouts
11

What vLLM Needs in the Sync World

Live weight updates turn inference into a stateful training component.

Weight update path

Receive new tensors into live workers without rebuilding the serving process.

Cache reset after weight changes

Prefix and KV cache state must not outlive the model weights that created it.

Lifecycle boundaries

Coordinate generation, pause, update, and resume so rollouts stay coherent.

vLLM engine training mode
generate rollouts
pause safe boundary
update new weights

The serving engine now needs correctness hooks. vLLM TRL and weight-transfer docs are the follow-up path.

Sync Fixes IO, But Creates Idle GPUs

Fresh weights are useful, but the work still runs one phase at a time.

rollout train rollout train
Trainer GPUs
idle waiting for samples
train policy update
idle waiting again
train next update
vLLM GPUs
generate rollouts
idle trainer owns step
generate fresh rollouts
idle trainer owns step
Fresh, but serialized

Inference and training still alternate instead of overlapping.

Slowest rollout sets the clock

Fixed batches wait for stragglers before the next update.

Tighter failure coupling

A worker failure can restart the whole synchronized run.

13

Async Loop: Keep Both Sides Busy

Overlap rollout generation and training with bounded staleness.

Continuous rollout production

vLLM keeps generating instead of waiting for trainer phases.

Controller buffers samples

Tasks and completed rollouts flow through queues.

Trainer consumes ready batches

Updates proceed when enough useful samples are available.

producer vLLM variable-length rollouts
scheduler + buffer Controller queues tasks and samples
consumer Trainer batches ready samples
no fixed batch barrier weights up to K steps old higher utilization
